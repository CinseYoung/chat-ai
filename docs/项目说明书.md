## 前言：先看全景

```
用户打开浏览器
    │
    ▼

    ▼
```

**逐行解读**：

| 写法 | 含义 |
|------|------|
| `func loadConfig() *Config` | 定义一个函数，名叫 `loadConfig`，返回一个 `Config` 的指针 |

### 53-83 行：main 函数（程序入口）

```go
func main() {
    cfg := loadConfig()                    // ① 加载配置
    store, err := NewStore(cfg)            // ② 初始化向量库
    if err != nil {                        // ③ 如果出错就退出

    log.Printf("[chat-ai] vector store ready, chunks=%d", store.Count())

    r := gin.Default()                     // ④ 创建 Gin 实例
    r.Use(cors.New(cors.Config{            // ⑤ 配置 CORS（允许跨域）

        AllowMethods:    []string{"GET", "POST", "OPTIONS"},
        AllowHeaders:    []string{"*"},

    r.Static("/assets", "static/assets")            // ⑧ 静态文件
```

**步骤拆解**：

| 步骤 | 做了什么 | 前端概念类比 |
|------|---------|-------------|
| ① | 加载配置（读环境变量 + 默认值） | `process.env.PORT` |
| ② | 初始化向量数据库（读磁盘上的 `.gob` 文件） | `new Database()` |
| ③ | 出错就退出，打日志 | `process.exit(1)` |
| ④ | 创建 Gin 框架实例 | `const app = express()` |
| ⑤ | 配置 CORS 中间件 | `app.use(cors())` |
| ⑥ | 注册三个 API 路由 | `app.get('/api/health', handler)` |
| ⑦ | 所有未匹配路由返回 index.html | SPA 配置 |
| ⑧ | `/assets/` 映射到 `static/assets/` 目录 | `express.static()` |
| ⑩ | 在指定端口启动服务 | `app.listen(8000)` |

**⑦⑧ 为什么需要**：部署时前端和后端在同一个端口。用户访问 `/` → 返回 `index.html`；HTML 里引用了 `/assets/xxx.js` → 从 `static/assets/` 目录读取。

#### 第一部分：准备阶段（126-145）

```go
func chatStream(cfg *Config, store *Store) gin.HandlerFunc {
    return func(c *gin.Context) {
        message := c.Query("message")          // 从 URL 参数读用户消息
        if message == "" {
            c.JSON(400, gin.H{"error": "message required"})
            return
        }
        useRAG := c.Query("use_rag") == "true" // 读 RAG 开关
```

`c.Query("message")` 就是从 `?message=你好&use_rag=true` 里取值。

```go
        // 设置 SSE 响应头
        c.Writer.Header().Set("Content-Type", "text/event-stream")
        c.Writer.Header().Set("Cache-Control", "no-cache")
        c.Writer.Header().Set("X-Accel-Buffering", "no")
```

这三个响应头告诉浏览器："我要给你发 **SSE 流**（Server-Sent Events），不要缓存，不要缓冲。"

**什么是 SSE？** 一种服务端主动推数据给浏览器的方式。普通的 HTTP 是"浏览器问一次，服务器答一次"；SSE 是"浏览器问一次，服务器持续答，直到说完"。

```go
        flusher, ok := c.Writer.(http.Flusher)
        if !ok {
            c.JSON(500, gin.H{"error": "streaming unsupported"})
            return
        }
```

SSE 需要服务器支持"分段发送"（Flush）。`flusher` 就是用来"手动推数据"的工具。如果服务器不支持，直接返回错误。

```go
        send := func(event string, data any) {
            payload, _ := json.Marshal(data)
            fmt.Fprintf(c.Writer, "event: %s\ndata: %s\n\n", event, payload)
            flusher.Flush()
        }
```

**这是最精妙的一行**：定义了一个 `send` 函数，以后只需要 `send("context", 数据)` 就能推一条 SSE 事件给前端。

SSE 格式：
```
event: context
data: {"hits": [...], "top_k": 3}

event: content
data: {"content": "我们"}

```

3. 拼接增强后的 Prompt：system 提示词（要求严格基于资料）+ 用户问题 + 参考资料

### 39-84 行：ChatStream 函数（流式对话）

```go
func ChatStream(ctx context.Context, baseURL, model string, messages []Message, onToken func(string) error) error {
```

这个函数把 messages 发给 Ollama，Ollama 返回流式响应。每收到一个 token，调用 `onToken` 回调。

```go
    body, _ := json.Marshal(map[string]any{
        "model":    model,
        "messages": messages,
        "stream":   true,                                   // 流式模式！
        "options":  map[string]any{"temperature": 0.3, "num_predict": 2048},
    })
```

**参数解释**：

| 参数 | 值 | 含义 |
|------|----|------|
| `model` | `qwen3:8b` | 用哪个模型 |
| `messages` | 对话历史 | 发给模型的完整上下文 |
| `stream` | `true` | **流式模式**：模型一边生成一边返回，不等全部生成完 |
| `temperature` | `0.3` | 随机性控制（0=完全确定，1=非常随机）。0.3 适合需要精确答案的场景 |
| `num_predict` | `2048` | 最多生成多少个 token |

**为什么要 `stream: true`？** 如果不流式，用户体验是：输入问题 → 等 10 秒 → 答案一次性出来。流式是：输入问题 → 答案一个字一个字出来，像 ChatGPT 那样。

```go
    reader := bufio.NewReader(resp.Body)
    for {
        line, err := reader.ReadBytes('\n')  // 读一行（Ollama 的流式响应是 NDJSON 格式）
```

Ollama 流式返回的是 **NDJSON**（每行一个独立 JSON）：

```json
{"model":"qwen3:8b","message":{"role":"assistant","content":"Phoenix"},"done":false}
{"model":"qwen3:8b","message":{"role":"assistant","content":"-7"},"done":false}
{"model":"qwen3:8b","message":{"role":"assistant","content":""},"done":true}
```

**`ReadBytes('\n')`**：读一行，以换行符结束。和前端 `readline()` 一个意思。

```go
        if len(line) > 0 {
            var chunk struct {
                Message Message `json:"message"`
                Done    bool    `json:"done"`
            }
            if json.Unmarshal(bytes.TrimSpace(line), &chunk) == nil {
```

把这一行 JSON 解析出来，拿到 `message.content`（生成的文本）和 `done`（是否结束）。

```go
                if chunk.Message.Content != "" {
                    if cberr := onToken(chunk.Message.Content); cberr != nil {
                        return cberr
                    }
                }
                if chunk.Done {
                    return nil  // 模型说完了
                }
```

如果有新 token → 回调 `onToken`，主函数里会 `send("content", ...)` 推给前端。如果 `done: true` → 结束循环。

```go
        if err == io.EOF {   // 正常结束
            return nil
        }
        if err != nil {      // 异常结束
            return err
        }
    }
```

三种退出方式：
- `done: true` → 模型主动结束
- `io.EOF` → 数据读完
- `err != nil` → 出错了

### 42-65 行：embedFn（向量化 + 归一化）

```go
func embedFn(cfg *Config) chromem.EmbeddingFunc {
    return func(ctx context.Context, text string) ([]float32, error) {
        v, err := Embed(ctx, cfg.OllamaBaseURL, cfg.EmbedModel, text)
        //  ↑ 调用 ollama.go 的 Embed 函数，把文本发给 Ollama
```

chromem-go 每次需要向量化文本时，就调这个函数。

```go
        // —— 下面是 L2 归一化 ——
        var sum float64
        for _, x := range v {
            sum += float64(x) * float64(x)    // 平方和
        }
        if sum == 0 {
            return v, nil
        }
        norm := float32(math.Sqrt(sum))       // 开根号 = L2 范数
        out := make([]float32, len(v))
        for i, x := range v {
            out[i] = x / norm                 // 每个元素除以范数
        }
        return out, nil
    }
}
```

**为什么需要 L2 归一化？**

Ollama 返回的向量没有归一化。不归一化的话，相似度分数在奇怪的范围（比如 0.5-2.3），无法直观判断"多相似算相似"。

归一化后，点积 = 余弦相似度，结果落在 [-1, 1]，1 表示完全相同。

**数学过程**（不用理解，大概知道就行）：
```
原始向量: [3, 4, 0]
平方和: 3² + 4² + 0² = 25
L2 范数: √25 = 5
归一化后: [3/5, 4/5, 0] = [0.6, 0.8, 0]
```

```go
func buildRAGUser(question string, hits []Hit) string {
    var sb strings.Builder                       // 高效拼接字符串的工具
    sb.WriteString("【参考资料】\n")
    for i, h := range hits {
        fmt.Fprintf(&sb, "[%d] (来源: %s, 相似度: %.3f)\n%s\n\n",
            i+1, h.Source, h.Similarity, h.Content)
    }
    sb.WriteString("\n【问题】\n")
    sb.WriteString(question)
    return sb.String()
}
```

## 总结：一次完整请求的全链路

以用户提问"我们组的代号是什么？"且打开 RAG 为例：

```
┌─ 浏览器 ─────────────────────────────────────────────────────────┐
│ useChat.sendMessage("我们组的代号是什么？", session, {useRAG:true}) │
│   ↓                                                               │
│ new EventSource("/api/chat/stream?message=...&use_rag=true")     │
└───────────────────────────────────────────────────────────────────┘
    │ SSE 连接
    ▼
┌─ Go 后端 main.go ────────────────────────────────────────────────┐
│ chatStream handler 收到请求                                        │
│   useRAG=true → 走 RAG 分支                                       │
│   ├─ rag.go: store.Query("我们组的代号是什么？", k=3)             │
│   │   └─ chromem-go 内部: 向量化查询 → 余弦相似度 → Top-3        │
│   ├─ SSE: 推送 "context" 事件（3条命中）                           │
│   ├─ rag.go: buildRAGUser() 拼增强 Prompt                         │
│   └─ ollama.go: ChatStream() 发给 Ollama                          │
│       ├─ SSE: 推送 "content" 事件（逐 token）                      │
│       └─ SSE: 推送 "done" 事件                                    │
└───────────────────────────────────────────────────────────────────┘
    │                                     │
    │   SSE: context                      │  POST /api/embeddings
    │   SSE: content (每个token)           │  POST /api/chat
    │   SSE: done                         ▼
    ▼                              ┌──────────────┐
┌─ 浏览器 ──────────────────────   │   Ollama     │
│ 渲染: "Phoenix-7 [1]"            │   :11434     │
│ 展开命中: team-handbook.md       └──────────────┘
└──────────────────────────────────
```
